# Vouch Backend - Team Workflow Plan

## Overview
3 backend developers working on FastAPI backend for AI-powered fact-checking.

## Division of Work

### Developer 1: API Layer & Architecture
**Focus**: Routes, schemas, project structure

**Responsibilities**:
- Define Pydantic models for requests/responses
- Set up API endpoints
- Handle request/response validationOk
- CORS and middleware configuration
- Error handling structure
- API documentation (auto-generated by FastAPI)

**Key Deliverables**:
```
backend/
├── main.py                 # Entry point (exists)
├── models/
│   ├── __init__.py
│   ├── schemas.py         # Pydantic models
│   └── enums.py           # Verdict enums, etc.
├── routes/
│   ├── __init__.py
│   └── analysis.py        # /api/analyze endpoints
├── middleware/
│   └── error_handler.py
└── config.py              # Settings, env vars
```

**Endpoints to Build**:
- `POST /api/analyze` - Submit claim for analysis
- `GET /api/analyze/{id}` - Get analysis result
- `GET /api/history` - Get recent analyses
- `GET /api/health` - Health check

---

### Developer 2: AI/ML Integration
**Focus**: LLM integration, claim analysis logic

**Responsibilities**:
- Integrate with Claude/OpenAI API
- Build analysis pipeline
- Prompt engineering for fact-checking
- Parse and structure LLM responses
- Confidence scoring algorithm
- Bias detection logic

**Key Deliverables**:
```
backend/
├── services/
│   ├── __init__.py
│   ├── llm_service.py      # LLM client wrapper
│   ├── analysis_service.py # Main analysis logic
│   └── prompt_templates.py # Prompts for different analyses
├── utils/
│   └── claim_parser.py     # Extract claims from text
└── core/
    └── analysis.py         # Analysis result builder
```

**Analysis Pipeline**:
1. Receive claim text
2. Query LLM for verdict (true/false/misleading/unverifiable)
3. Query LLM for explanation
4. Query LLM for bias/framing analysis
5. Query LLM for simplified explanation
6. Combine into structured response

---

### Developer 3: Data Sources & Research
**Focus**: External APIs, web scraping, source gathering

**Responsibilities**:
- News API integration (NewsAPI, GDELT, etc.)
- Web search integration (Serper, Google Custom Search)
- Source credibility scoring
- Web scraping fallback
- Caching layer for sources
- Source metadata extraction

**Key Deliverables**:
```
backend/
├── services/
│   ├── news_service.py      # News API client
│   ├── search_service.py     # Web search client
│   └── source_scorer.py      # Credibility scoring
├── scrapers/
│   ├── __init__.py
│   └── article_scraper.py   # Fallback scraping
├── cache/
│   └── redis_client.py      # Simple caching
└── data/
    └── trusted_sources.json # Known reliable sources list
```

**Source Pipeline**:
1. Search for relevant articles about claim
2. Filter by source credibility
3. Extract summaries and metadata
4. Return top 3 sources with analysis

## Integration Points

### 1. Shared Schema (Developer 1 leads, all review)
```python
class AnalysisRequest(BaseModel):
    claim: str
    context: Optional[str] = None

class AnalysisResponse(BaseModel):
    id: str
    verdict: Literal["true", "false", "misleading", "unverifiable"]
    trust_score: int  # 0-100
    confidence: int   # 0-100
    explanation: str
    why_misleading: Optional[str]
    bias_analysis: Optional[str]
    core_claim: str
    sources: List[Source]
    simple_explanation: Optional[str]
```

### 2. Service Interface (All agree on)
```python
# services/analysis_orchestrator.py
async def analyze_claim(claim: str) -> AnalysisResult:
    # Developer 2: LLM analysis
    # Developer 3: Source gathering (parallel)
    # Combine results
    pass
```

### 3. Development Order
1. **Hour 0-1**: Developer 1 sets up project structure, all review schemas
2. **Hour 1-3**: Parallel development on each module
3. **Hour 3-4**: Integration testing
4. **Hour 4-6**: Polish, error handling, edge cases
5. **Hour 6+**: Performance optimization, caching

## Communication Plan

- **Standup every 2 hours** - Share blockers, progress
- **Shared schema is sacred** - No breaking changes without team approval
- **Test with curl/Postman** before declaring done
- **Document env vars** in shared .env.example
